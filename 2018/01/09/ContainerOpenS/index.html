黄金镜像

我对容器感兴趣的原因之一是，容器允许将具有所有依赖项的应用程序打包到单个部署单元中，这个部署单元称为黄金镜像，方便从一个环境迁移到另一个环境。例如，您可以将其从集成 (integration) 迁移到 UAT (User Acceptance Test，用户验收测试) 或从准上线 (staging) 迁移到生产 (production)。因为单个单元中部署了所有依赖关系，所以可以保证前一个单元的内容在后一个单元仍然可以运行。

在容器崛起之前，很多公司困于应用推广上，其中有些人为此写了详细的安装步骤说明，应用必须手动安装和严格遵守安装步骤，另外的人则为此投入了大量的精力开发 Puppet ， Chef 或 Python 编写软件自动化构建脚本，此外，除了初始的投入，还需要大量的维护工作来应对软件的升级和变更，大多数人都曾遇到过各种各样的问题。容器带来的是一个简单的标准化方法，应用程序的依赖关系（操作系统，运行时环境（例如：JVM (Java Virtual Machine, Java虚拟机)），库以及一些配置）是容器镜像的一部分，通过启动容器镜像实例，可以将应用程序部署于不同的环境。

可重用性

在以后构建相同镜像会产生相同的结果，这对于获得可用于补丁、升级和进一步演进的稳定参考非常重要，这要求用于创建容器映像的依赖项（库或其他镜像）被唯一引用和版本化。

敏捷的标准运行环境

​

同质化的企业软件环境具有以下鲜明的优势：

  • 通过限制可能需要支持的软件运行环境的数量，来节省维护的时间和成本。
  • 方便搭建适用于业务部门的软件运行环境，这对于及时解决突发事件是至关重要的。
  • 在几个不同的环境下，建立合规 (Compliance) 更容易。
  • 方便工作人员更容易，因为软件运行环境相同，有利于员工适应岗位地变更。

在定义镜像格局时，应该考虑上述优势带来的便利性，有时严格执行和缓慢发展 SOE (Server Object Extension, 服务对象扩展), 服务对象扩展)会造成企业反应迟缓，容器镜像可以帮助解决高可重用性和方便对多个目标进行集中更改的问题。

可重用性

第一次创建 SOE (Server Object Extension, 服务对象扩展) 时，限制能被组件注入的框架、应用程序服务器、驱动程序和脚本等的镜像的数量是非常重要的，这些可以用于基础功能、连接性、监听、资源跟踪管理、安全性等。分层的方法的继承和组合可能支持这一点，在这方面，重点考虑全局镜像而不是单一镜像。

可维护性

上一节提到的中央注入点也是修补和升级的关键，这些更改（可能由软件提供商作为容器镜像提供）需要自动级联，以便维护变得简单。

在前容器时代，我看到很多公司都在努力保持Java或应用服务器的拥有最新补丁和进行安全修复，容器技术具备在中心位置应用变更的可能性，并使容器在软件环境中推出既不费力，也不需要停机支持，这在安全性和可靠性方面会产生巨大的影响，因此，镜像设计时需要考虑可维护性。

资源利用效率

在创建应用程序时，资源（RAM，CPU，存储等）的尽量小的消耗是显而易见的目标，因此使用镜像进行包装，这意味着密度更高，成本更低。与虚拟机相比，容器的优势是具有共享相同内核而不是创建额外实例，这可以通过容器共享层来进一步推动，遵循已经提到的一些目标，这让SOE (Server Object Extension, 服务对象扩展) 和可重用成为可能。尽管容量大小确实影响推拉镜像所需的时间，但它在最终的 RAM 和存储使用方面起着比镜像容量本身更大的作用。

安全

除了保持操作系统，应用程序服务器和其他库补丁及时修补保证最新的安全性之外，在创建映像时还需要考虑安全性方面的一些其他重要方面：

  • 减少攻击面：这可以通过将镜像内部安装的内容限制在运行容器所需的内容来完成。
  • 限制功能：特权容器应该是执行特定安全措施的例外。其他功能，如安装主机文件系统，绑定主机端口，或使用可能与主机上的用户相匹配的特定用户标识运行，应仅在没有其他选项可用时才被授权，并且在仔细检查安全性影响风险之后得到缓解。
  • 有限的资源：容器映像应该以能够在有限资源（ CPU ， RAM ，网络，存储）下运行的方式构建，以便它不会成为拒绝服务攻击主机和其他容器在同一台主机上运行。

弹性

像 Kubernetes 和 OpenShift 这样的PaaS (Platform-as-a-Service, 平台即服务) 平台提供了监控和自我修复机制、就绪和容器监测探针 (Liveness Probes)确保以下行为：

  • 当容器没有准备好时，请求不会被发送到容器，实际上，这意味着容器没有完成启动，容器正在终止或者它的一个依赖项（数据库，服务）不可用。
  • 容器监测探针 (Liveness Probes) 失败时，容器将重新启动，重新启动后容器将恢复到初始状态，类似于返厂重置。

另一个方面是，当 OpenShift 想要终止一个容器时，它首先将容器从请求处理旋转中移出并发送一个 SIGTERM 信号，它为应用程序在结束之前正常关闭提供了时间，如果允许的期限已过，则使用 SIGKILL 信号，在这方面，镜像内的应用程序应完成处理正在进行的请求，释放资源，并在接收到 SIGTERM 信号时终止。

易于使用

可复用性经常被作为一个目标提及，但是，只有在镜像易于使用的情况下才能达到较好的水平，其中包括以下几个方面：

  • 可以通过继承，组成或利用扩展点来进行扩展，但是扩展点需要首先考虑，作为一名镜像设计师，您有责任思考如何以简单的方式添加功能或自定义镜像。
  • 文档：这包括记录镜像的目的，使用情况以及运行它的重要方面，例如它的入口点，使用的端口，应用程序可以写入数据的目录等，镜像快速启动也是一种有利于终端用户了解应该如何使用镜像的方法。
